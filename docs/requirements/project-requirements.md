# 心灵情绪日记项目需求文档

项目名称：心灵情绪日记 / Mindful Memory Diary / Inner Garden

## 1. 项目背景

在日常学习和生活压力中，很多用户有记录情绪和回顾自我的需求，但传统日记存在三个常见问题：不知道从哪里开始写、难以坚持、写完后缺少反馈和整理。尤其是在情绪比较混乱的时候，用户往往只想先被温柔地听见，而不是立刻写出一篇完整日记。

心灵情绪日记希望用 AI 对话降低记录门槛。用户可以输入文字、上传图片，或用一句很短的话开始表达；AI 以陪伴者的方式进行回应和轻柔追问，并在对话结束后帮助用户整理成一篇结构化日记。日记会进一步转化为可保存、可回顾的记忆卡片，沉淀到 Memory Garden 中，形成属于用户自己的情绪记忆花园。

本项目当前仓库定位为课程团队协作项目：前端目录为 React + Vite 骨架，后端目录为 FastAPI 骨架，文档放在 `docs/requirements/` 和 `docs/design/` 中。本阶段先完成需求、页面和接口设计文档，为后续前后端实现提供明确依据。

## 2. 核心痛点

1. 用户想记录情绪，但不知道该写什么。
2. 用户情绪低落或疲惫时，不愿意写长篇日记。
3. 普通日记只有记录，没有陪伴式反馈，用户难以获得继续表达的动力。
4. 历史日记通常以列表方式展示，回忆感和沉浸感不足。
5. 用户希望回看过去的自己，但缺少一种温柔、低压力的复盘方式。

## 3. 核心价值

1. 降低记录门槛：用户可以从一句话或一张图片开始，不需要一开始就写完整日记。
2. 提供陪伴体验：AI 用温柔、克制、非评判的语言回应用户，帮助用户慢慢说清楚今天发生了什么。
3. 自动整理内容：系统根据对话生成标题、日期、正文、情绪总结和记忆摘要。
4. 可视化保存回忆：日记被保存成记忆卡片，形成 Memory Garden，而不是普通文本列表。
5. 支持未来回看：用户可以进入某一张记忆卡片，查看当时的情绪、封面图和完整日记，并与“基于这篇日记生成的过去自我画像”对话。

## 4. 产品定位

心灵情绪日记是一个 AI 陪伴式情绪记录网页产品。它不是心理诊断工具，也不是治疗工具，而是帮助用户完成“倾诉、整理、保存、回看”的日记记录工具。

产品关键词：

- AI 温柔陪伴
- 情绪日记
- 记忆卡片
- Memory Garden
- 过去自我对话
- 治愈系视觉体验

一句话定位：

> 一个通过 AI 温柔陪伴用户倾诉、整理情绪，并把每天的心情沉淀成可视化记忆卡片的情绪日记网页产品。

## 5. 目标用户

主要面向以下用户：

1. 想记录情绪，但不想写长篇日记的人。
2. 每天压力较大，希望有一个安静倾诉空间的学生或年轻用户。
3. 希望有人用温柔方式回应自己，而不是被评价或说教的人。
4. 想回顾过去心情、理解自己变化的人。
5. 喜欢图像化、卡片化、花园式记录体验的人。

## 6. 产品边界

本产品只做情绪记录、陪伴和回忆整理，不提供心理诊断、心理治疗、医疗建议或危机干预结论。

明确边界：

1. 不判断用户是否存在心理疾病。
2. 不输出诊断名称、治疗方案、用药建议或医疗结论。
3. 不承诺解决用户心理问题。
4. 不替代心理咨询师、医生或其他专业人士。
5. AI 回复应保持陪伴、倾听、记录整理的角色，不进行强指导或绝对化判断。
6. 当用户表达明显危险或紧急情况时，系统应提示用户联系现实中的可信任的人或当地紧急求助渠道，但不做诊断。

## 7. MVP 核心流程

MVP 需要跑通以下核心闭环：

1. 用户进入首页。
2. 点击“开始今天的记录”。
3. 进入 AI Companion Chat。
4. 用户输入今天发生的事情，或上传一张图片。
5. AI 进行温柔回应和轻柔追问。
6. 用户点击“我说完了，生成日记”。
7. AI 根据对话生成结构化日记。
8. 用户选择情绪和情绪底色，可编辑日记内容。
9. 如果用户没有上传图片，AI 生成一张风格化封面图。
10. 用户点击“保存到 Memory Garden”。
11. Memory Garden 展示新增记忆卡片。
12. 用户点击记忆卡片，进入 Memory Detail。
13. 用户查看完整日记，并可与“过去的我”进行基于该日记的对话。

## 8. MVP 必须功能

### 8.1 首页

- 展示产品名称和一句清晰的产品说明。
- 展示产品边界提示：不是心理诊断工具，只陪伴记录和整理情绪。
- 提供“开始今天的记录”按钮。
- 提供“进入 Memory Garden”按钮。

### 8.2 AI 对话

- 支持文字输入。
- 支持图片上传作为今日记忆素材。
- 支持 AI 温柔回复。
- 支持 AI 轻柔追问。
- 支持“换个问题问我”，避免用户被单一问题卡住。
- 支持“我说完了，生成日记”结束对话。

### 8.3 日记生成

- 根据对话生成结构化日记。
- 日记至少包含标题、日期、正文、今日情绪、情绪总结、记忆摘要。
- 用户可以编辑 AI 生成的日记文本。
- 用户可以选择情绪标签。
- 用户可以选择情绪底色。
- 如果没有上传图片，系统生成一张风格化封面图。
- 支持保存到 Memory Garden。

### 8.4 Memory Garden

- 展示用户保存过的所有记忆卡片。
- 每张卡片展示封面图、日期、标题和主要情绪。
- 支持按时间排序。
- 点击卡片进入 Memory Detail。
- 提供“写下今天”入口。

### 8.5 Memory Detail / Past Self Chat

- 展示完整日记内容。
- 展示封面图、日期、情绪和关键词。
- 支持开启“和那天的我聊聊”。
- AI 根据该日记内容，以“过去自我画像”的方式进行对话。
- 支持返回 Memory Garden。

### 8.6 管理员基础能力

- 管理员可查看基础统计数据，例如总记忆卡片数、总对话数、近 7 天新增记录数。
- 管理员可查看系统运行状态和接口是否可用。
- MVP 管理员端不查看用户隐私日记正文，避免超出课程项目和产品边界。

## 9. MVP 暂不做功能

1. 语音输入和语音合成。
2. 复杂情绪趋势分析。
3. 月度情绪报告或周报。
4. 账号系统和多设备同步。
5. 社交分享。
6. 高级图片编辑。
7. 复杂搜索。
8. AI 主动推送提醒。
9. 专业心理测评、心理诊断、治疗建议。
10. 复杂管理员审核后台。

## 10. 用户端和管理员端区别

### 10.1 用户端 User

用户端关注个人记录体验，主要功能是：

- 开始今日情绪记录。
- 与 AI 陪伴者对话。
- 上传图片或输入文字。
- 生成、编辑并保存日记。
- 浏览 Memory Garden。
- 查看记忆详情。
- 与“过去的我”进行复盘式对话。

用户端强调隐私感、陪伴感和视觉治愈感。

### 10.2 管理员端 Admin

管理员端关注课程项目演示和系统维护，主要功能是：

- 查看基础统计：对话数量、日记数量、记忆卡片数量、近 7 天新增记录。
- 查看服务状态：前端可访问、后端健康检查、AI 接口配置状态。
- 查看功能使用概览：用户更常使用文字输入、图片上传或记忆详情。

管理员端不作为心理咨询后台，不查看或评判用户心理状态，不提供用户诊断结果。

## 11. 验收标准

### 11.1 产品闭环验收

- 用户可以从首页进入 AI 对话页。
- 用户可以完成至少一轮文字输入和 AI 回复。
- 用户可以点击生成日记，并看到结构化日记结果。
- 用户可以选择情绪和情绪底色。
- 用户可以保存日记为记忆卡片。
- Memory Garden 可以展示刚保存的记忆卡片。
- 用户点击记忆卡片后可以进入详情页。
- 用户可以在详情页开启与“过去的我”的对话。

### 11.2 内容边界验收

- 首页或关键页面展示“不是心理诊断工具”的边界说明。
- AI 回复不包含诊断、治疗、用药或医疗建议。
- 过去自我对话明确基于该日记内容生成，不冒充真实用户本人。
- 情绪总结使用“记录、理解、回看”的表达，不使用医学诊断表达。

### 11.3 页面验收

- 至少完成 5 个页面或页面状态：Home、AI Companion Chat、Diary Result / Mood Check-in、Memory Garden、Memory Detail / Past Self Chat。
- 每个页面有明确主按钮和跳转路径。
- 页面文案适合课程项目答辩，表达清楚产品价值和功能边界。

### 11.4 数据验收

- 一张记忆卡片至少保存标题、日期、正文、情绪、情绪底色、封面图、对话摘要。
- Memory Garden 列表数据来自已保存的记忆卡片。
- Memory Detail 能根据卡片 ID 获取对应详情。

### 11.5 管理员验收

- 管理员可以查看基础统计数据。
- 管理员统计不包含心理诊断类字段。
- 管理员统计不直接暴露用户完整私密日记正文。

本节用于把前文产品需求进一步落到《系统设计与实践》课程验收要求上。以下内容是对原有需求的课程化补充，不改变产品定位、用户场景、功能边界和 MVP 主流程。

前文“账号系统和多设备同步”属于 MVP 暂不做功能，指的是不做复杂商业化账号体系、多设备同步、找回密码、第三方登录等完整账户能力。

## 13. 基础栈实现方案

### 13.1 前后端分离

- 前端：React + Vite。
- 后端：FastAPI。
- 前端通过 HTTP API 请求后端数据。
- 后端统一返回 JSON 数据。
- 页面展示的数据不使用静态截图代替，核心列表、详情、统计图都来自后端接口。

### 13.2 数据库或结构化数据

MVP 建议使用 SQLite。数据对象至少包含以下几类，满足课程对“至少 2 类数据对象，且数据之间有关联”的要求。

| 数据对象 | 作用 | 主要关联 |
| --- | --- | --- |
| users | 保存演示用户和管理员账号 | 一个用户可以拥有多条对话、日记和记忆卡片 |
| conversations | 保存一次今日记录对话的基本信息 | 关联 users |
| messages | 保存用户输入和 AI 回复 | 关联 conversations |
| diary_entries | 保存 AI 整理后的结构化日记 | 关联 users 和 conversations |
| memory_cards | 保存 Memory Garden 中展示的记忆卡片 | 关联 diary_entries |
| ai_outputs | 保存 AI 生成结果，例如情绪总结、封面图提示词、过去自我回复 | 关联 diary_entries 或 memory_cards |

最低数据操作要求：

- 新增：新增对话消息、结构化日记、记忆卡片。
- 查询：查询 Memory Garden 列表、记忆详情、管理员统计数据。
- 修改：用户编辑 AI 生成的日记内容、情绪标签或情绪底色。
- 删除：用户或管理员可删除课程演示用测试记忆卡片；建议采用软删除，避免误删真实记录。

### 13.3 登录与角色区分

项目至少提供普通用户和管理员两类角色。

| 角色 | 可见功能 | 不可见或受限功能 |
| --- | --- | --- |
| 普通用户 user | 首页、AI 对话、日记生成、Memory Garden、Memory Detail、Past Self Chat | 不进入管理员统计页 |
| 管理员 admin | 管理员统计页、系统健康检查、功能使用概览、演示数据管理 | 不直接查看用户完整私密日记正文，不做心理评判或诊断 |

实现要求：

- 登录成功后，前端根据角色展示不同入口。
- 后端接口根据角色进行权限判断。
- 密码字段保存为 `password_hash`，不保存明文密码。
- 演示账号只用于课程展示，不应提交真实个人密码。

### 13.4 动态数据可视化

MVP 不做复杂情绪趋势分析、周报或月报，但需要完成至少一个来自后端真实数据的动态图表。

建议优先实现管理员端基础统计图：

- 近 7 天新增记忆卡片数量折线图。
- 主要情绪分布柱状图或饼图。
- 输入方式分布图，例如文字输入、图片上传。

可使用 ECharts、Chart.js 或 Recharts。图表数据必须来自后端统计接口；当新增、修改或删除记忆卡片后，刷新页面或点击按钮，图表数据应发生变化。

## 14. 特色技术深度集成方案

本项目选择“AI 模型 API 集成”作为特色技术。AI 不是右下角独立聊天按钮，而是直接参与情绪记录主流程。

### 14.1 AI 深度集成位置

| AI 功能 | 所在页面 | 对系统的影响 |
| --- | --- | --- |
| 温柔陪伴式回复 | AI Companion Chat | 影响用户是否继续表达和补充记录 |
| 轻柔追问 | AI Companion Chat | 帮助用户把碎片表达补充成可整理内容 |
| 结构化日记生成 | Diary Result / Mood Check-in | 生成标题、正文、情绪总结、记忆摘要 |
| 封面图提示词或封面图生成 | Diary Result / Memory Garden | 影响记忆卡片视觉展示 |
| 过去自我对话 | Memory Detail / Past Self Chat | 基于单篇日记生成复盘式对话 |

### 14.2 AI 深度集成验收点

- AI 输出会影响页面展示，例如日记正文、情绪总结、封面图提示词、记忆摘要和过去自我回复。
- 项目有自定义提示词设计，不只是简单问答。
- AI 调用结果会保存到数据库，并参与 Memory Garden、Memory Detail 或管理员统计展示。
- 如果现场 API 不稳定，应准备备用演示方案，例如 mock AI 返回、样例数据或录屏。

### 14.3 AI 提示词分类

项目至少保留以下几类提示词，便于 Vibe Coding 日志和答辩说明：

1. 陪伴式对话提示词：要求 AI 温柔、克制、非评判地回应用户。
2. 结构化日记生成提示词：把对话整理为标题、日期、正文、今日情绪、情绪总结和记忆摘要。
3. 封面图生成提示词：根据日记内容生成治愈系水彩、淡油画或梦境花园风格封面图描述。
4. 过去自我对话提示词：明确 AI 只基于该日记生成“过去自我画像”，不冒充真实用户本人。
5. 内容边界提示词：限制 AI 不输出心理诊断、治疗方案、用药建议或医疗结论。

## 15. 核心 API 草案

以下接口用于指导后续开发。实际实现时可以根据代码结构调整命名，但要保证前端、后端和数据流能够跑通。

| 模块 | 方法 | 路径 | 权限 | 作用 |
| --- | --- | --- | --- | --- |
| 健康检查 | GET | `/api/health` | 公开 | 检查后端是否可用 |
| 注册 | POST | `/api/auth/register` | 公开 | 创建普通用户账号，课程演示可选 |
| 登录 | POST | `/api/auth/login` | 公开 | 返回当前用户身份和角色信息 |
| 当前用户 | GET | `/api/auth/me` | user/admin | 获取当前登录用户信息 |
| 对话消息 | POST | `/api/chat/messages` | user | 保存用户输入并返回 AI 回复 |
| 换个问题 | POST | `/api/chat/next-question` | user | 生成新的轻柔追问 |
| 生成日记 | POST | `/api/diaries/generate` | user | 根据对话生成结构化日记草稿 |
| 保存日记 | POST | `/api/diaries` | user | 保存用户确认后的日记 |
| 编辑日记 | PATCH | `/api/diaries/{diary_id}` | user | 修改日记正文、情绪标签或情绪底色 |
| 记忆卡片列表 | GET | `/api/memories` | user | 获取 Memory Garden 卡片列表 |
| 记忆卡片详情 | GET | `/api/memories/{memory_id}` | user | 获取 Memory Detail 页面数据 |
| 删除测试卡片 | DELETE | `/api/memories/{memory_id}` | user/admin | 删除或软删除课程演示用记忆卡片 |
| 过去自我对话 | POST | `/api/memories/{memory_id}/past-self-chat` | user | 基于该记忆卡片生成复盘式回复 |
| 管理员统计 | GET | `/api/admin/stats/overview` | admin | 获取总记忆卡片数、总对话数、近 7 天新增记录数 |
| 统计图数据 | GET | `/api/admin/stats/charts` | admin | 获取图表所需的后端统计数据 |

## 18. 课程验收脚本

演示时建议按以下顺序展示，保证老师能看到完整数据流。

1. 打开项目 README，说明前端、后端、数据库和启动方式。
2. 启动后端 FastAPI，打开 `/api/health` 展示后端可用。
3. 启动前端 React + Vite，进入 Home 页面。
4. 使用普通用户账号登录，展示普通用户入口。
5. 进入 AI Companion Chat，输入一段今日心情或上传一张图片。
6. 展示 AI 温柔回复和轻柔追问。
7. 点击“我说完了，生成日记”，展示结构化日记草稿。
8. 修改情绪标签或情绪底色，保存到 Memory Garden。
9. 进入 Memory Garden，展示刚刚新增的记忆卡片来自后端数据。
10. 点击记忆卡片进入 Memory Detail，展示完整日记和过去自我对话。
11. 切换管理员账号，进入 Admin Dashboard。
12. 展示统计图数据来自后端；新增或删除测试卡片后刷新图表，说明数据发生变化。
13. 展示 Vibe Coding 日志，讲清楚至少 1 次关键迭代。
14. 小组成员分别解释产品逻辑、前端交互、后端接口和数据表关系。
